Tài Liệu Hướng Dẫn AWKit

Chào mừng bạn đến với dự án Antigravity đã được khởi tạo!

🚀 Bắt Đầu (Getting Started)

AWKit (Antigravity Kit) là công cụ điều phối dự án, tạo code, và quản lý AI agent chuyên sâu. Dự án của bạn đã được khởi tạo với các file quan trọng sau đây:

  • .project-identity: Chứa cấu hình dự án (Tên, Tech Stack, Bundle ID). Hãy chỉnh sửa file này để thay đổi định nghĩa gốc của dự án.
  • CODEBASE.md: Tài liệu sống về kiến trúc của bộ code. Mở chat AI và gõ /codebase-sync để AI tự động đánh giá và cập nhật.
  • AGENTS.md & CLAUDE.md: Các tệp định tuyến (Router files) hỗ trợ AI IDE (như Cline, Roo, Cursor) nhận diện bộ quy tắc tập trung từ hệ thống.

⚡ Cách Sử Dụng AWKit

AWKit hoạt động theo nguyên tắc "nói gì làm nấy". Bạn chỉ cần mô tả yêu cầu bằng ngôn ngữ tự nhiên trong cửa sổ Chat, AI sẽ tự động kích hoạt đúng quy trình (7-Gate) mà không cần gõ lệnh thủ công.

🤖 AI Tự Động Xử Lý (Không cần gõ lệnh)

Các quy trình sau được hệ thống Skill tự động nhận diện và kích hoạt khi bạn chat:

  • Brainstorm: "Tôi muốn làm tính năng X" → AI tự chạy Gate 1 (Spec)
  • Thiết kế: "Thiết kế kiến trúc cho module Auth" → AI tự chạy Gate 2 (Architecture)
  • Code: "Code màn hình đăng nhập" → AI tự chạy Gate 4 (Execution) với 3-Phase
  • Debug: "App crash khi bấm nút Save" → AI tự kích hoạt systematic debugging
  • Refactor: "Đổi tên class UserManager" → AI dùng GitNexus rename an toàn

🧠 GitNexus - Trí Tuệ Mã Nguồn

AWKit tích hợp GitNexus để hiểu toàn bộ kiến trúc, rà soát mô hình tham chiếu (blast radiuses), và vẽ luồng hoạt động mã nguồn theo dạng đồ thị.

Cách dùng CLI (Terminal)

# Quét và đánh index lại dự án (Nên chạy sau mỗi lần commit lớn)
npx gitnexus analyze

# Quét đầy đủ bao gồm phân tích Embedding
npx gitnexus analyze --embeddings

Sử dụng với AI (MCP Tools)

Mọi AI Agent đi kèm AWKit đều có nhiệm vụ dùng các GitNexus Tool để tính toán rủi ro:

  • gitnexus_impact: Kiểm tra "blast radius" – Phải báo cáo xem có file nào hỏng không trước khi thay đổi (Refactoring).
  • gitnexus_query: Để lục tìm concept (e.g. "Tìm thư mục chứa Auth Flow").
  • gitnexus_detect_changes: Rà soát xem git local thay đổi những gì, có rủi ro nào ở scope khác không.
  • gitnexus_rename: Đổi tên một class hoặc biến an toàn thông qua Knowledge Graph thay vì chỉ Find and Replace.

📋 Symphony - Quản Lý Tiến Độ Tác Vụ

Symphony là trái tim của việc chia nhỏ và đánh dấu tiến độ công việc trong AWKit. Dữ liệu được lưu trữ ở cơ sở dữ liệu toàn cục (~/.gemini/antigravity/symphony/), không nằm trong thư mục dự án. Thư mục .symphony/ trong project chỉ đóng vai trò đánh dấu (marker) để AI nhận diện phạm vi dự án.

Cài đặt Symphony CLI (chỉ cần 1 lần):

# Cài đặt toàn cục
npm install -g @leejungkiin/awkit-symphony

# Kiểm tra đã cài xong
symphony status

Các thao tác CLI hữu ích:

# Hệ thống quản lý
symphony status                     # Xem thông tin cấu hình và trạng thái server
symphony reset                      # Xoá database toàn cục (CHỈ KHI BỊ LỖI DATABASE)

# Thao tác với Task
symphony task list                  # Xem danh sách tất cả các task hiện tại
symphony task add "Làm màn hình Đăng Nhập"
symphony task claim <id>            # Bắt đầu làm task
symphony task complete <id>         # Đánh dấu hoàn thành
symphony task block <id> "Lý do"   # Đánh dấu bị chặn vì lỗi

Hệ thống AI AWKit sẽ luôn tự động thực hiện các hành động trên cho bạn trong quá trình thực thi hệ thống 7-Gate. Bạn không cần gõ lệnh thủ công trừ khi muốn kiểm tra tiến độ.

🤖 Hệ Thống 7 Cổng (7-Gate) Tự Động

Bất cứ khi nào bạn nhờ AWKit làm một tác vụ có độ phức tạp cao, AI sẽ chạy theo quy trình vòng kín 7-Gate khắt khe này để kiểm soát chất lượng:

  1. Gate 1 (Spec): Thu thập yêu cầu, tạo file BRIEF.md
  2. Gate 1.5 (Module Spec): Viết sâu vào từng Flow người dùng và Rules validation trước khi code.
  3. Gate 2 (Architecture): Thống nhất Database Schema, API Contract.
  4. Gate 2.5 (Visual): Tạo mock-up UI với hệ thống tích hợp Pencil MCP hoặc ảnh chụp màn hình.
  5. Gate 3 (Tasks): Xé rách dự án thành các thẻ Symphony Tickets.
  6. Gate 4 (Execution): Thực thi theo 3-Phase (Code Kiến trúc chuẩn → Làm giao diện → Gắn Code Logic Data) với các bước Dừng User Test.
  7. Gate 5 (Verification): Review code, test và dùng AI xác nhận Git Commit an toàn.

🧩 Chiến Lược Đa IDE & Đa AI Model

AWKit được thiết kế để tận dụng sức mạnh của nhiều AI Model cùng lúc. Hệ thống gồm 3 Model chính, mỗi cái phục vụ một mục đích khác nhau, chia làm 2 kênh truy cập:

Kênh 1: Antigravity (IDE tích hợp — VS Code / Cursor)

Đây là kênh chính để code, debug, và điều phối dự án. Hoạt động qua Agent trong IDE.

Model Thế mạnh Khi nào dùng
Claude Opus 4.6 Mặc định Suy luận sâu, tuân thủ quy tắc tốt nhất, kiến trúc phức tạp Mọi task chính: code Feature, Debug logic phức tạp, Thiết kế kiến trúc, Gate 1–5
Gemini 3.1 Pro Bổ trợ Context window cực lớn (2M tokens), xử lý nhanh, phân tích file dài Đọc/phân tích codebase lớn, Refactor đa file, Brainstorm ý tưởng, Review nhanh

💡 Mẹo: Chuyển đổi model trực tiếp trong Antigravity bằng nút chọn Model ở thanh chat. Dùng Opus cho task cần chính xác cao, dùng Gemini khi cần quét/phân tích khối lượng lớn.

Kênh 2: Codex CLI / Extension (chạy nền song song)

Codex (GPT 5.4) hoạt động như một "trợ lý thứ hai" chạy song song, hoàn toàn READ-ONLY. Codex không chỉnh sửa code — nó chỉ đọc, phân tích, và xuất báo cáo.

Cách dùng Mô tả
Codex CLI Chạy trong Terminal: codex rồi nhập câu hỏi. Phù hợp để debug, review logic, và audit code.
VS Code Extension Cài extension "Codex" trong VS Code để dùng trực tiếp trong IDE song song với Antigravity. 
# Ví dụ dùng Codex CLI kiểm tra logic
codex "Review function validateBloodPressure và tìm edge case bị bỏ sót"

# Hoặc AWKit tự gọi Codex khi cần second opinion
/conductor-codex "Rà soát toàn bộ authentication flow"

Chiến lược tối ưu khi dùng đa model:

  1. Opus 4.6 dẫn đầu: Mọi task chính (code, debug, plan) đều chạy trên Opus. Đây là model có khả năng tuân thủ quy tắc AWKit và suy luận chuỗi tốt nhất.
  2. Gemini 3.1 Pro mở rộng: Khi cần đọc file dài, phân tích codebase >1000 dòng, hoặc brainstorm nhanh — chuyển sang Gemini để tận dụng context window lớn và tốc độ.
  3. Codex chạy song song: Mở terminal thứ 2, chạy codex để hỏi second opinion, review code đã viết, hoặc tìm bug mà model chính bỏ sót. Codex dùng quota riêng nên không ảnh hưởng Opus/Gemini.
  4. Không trùng lặp: Chỉ 1 model chỉnh sửa file tại 1 thời điểm. Codex chỉ đọc và báo cáo, không ghi file.
